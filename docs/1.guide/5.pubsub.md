# Pub / Sub

crossws supports native pub-sub API integration. A [peer](/guide/peer) can be subscribed to a set of named channels using `peer.subscribe(<name>)`. Messages can be published to a channel using `peer.publish(<name>, <message>)`. When publishing to a channel, the peer that does the publish is excluded from the broadcast so it won't receive it's own message.

```js
import { defineHooks } from "crossws";

const hooks = defineHooks({
  upgrade(req) {
    return {
      // namespace: new URL(req.url).pathname
    };
  },
  open(peer) {
    // Send welcome to the new client
    peer.send("Welcome to the server!");

    // Join new client to the "chat" channel
    peer.subscribe("chat");

    // Notify every other connected client
    peer.publish("chat", `[system] ${peer} joined!`);
  },

  message(peer, message) {
    // The server re-broadcasts incoming messages to everyone
    peer.publish("chat", `[${peer}] ${message}`);
  },

  close(peer) {
    peer.publish("chat", `[system] ${peer} has left the chat!`);
    peer.unsubscribe("chat");
  },
});
```

## Sync Backplane

> [!IMPORTANT]
> Experimental. The `sync` option and `crossws/sync` API may change between 0.x versions.

crossws pub/sub is in-memory and **local to one instance**. When you run more than one instance — multiple regions, processes, or replicas behind a load balancer — a `peer.publish("chat", …)` only reaches the peers connected to _that_ instance. A subscriber connected elsewhere never sees it.

A **sync adapter** bridges that gap: it relays published messages between instances over a shared backplane (e.g. Redis pub/sub) and fans inbound messages out to local subscribers. No change to your hooks and `subscribe` / `publish` keep working, they just span the cluster now.

```js
import nodeAdapter from "crossws/adapters/node";
import { broadcastChannelSync } from "crossws/sync";

const ws = nodeAdapter({
  hooks,
  sync: broadcastChannelSync(), // ← opt-in. Absent = today's local behavior.
});
```

## Drivers

### `broadcastChannelSync()`

Zero-dependency, built on [`BroadcastChannel`](https://developer.mozilla.org/en-US/docs/Web/API/BroadcastChannel).

Bridges instances that share a `BroadcastChannel` implementation (worker threads / processes on one host). Great for single-host fan-out.

### `redisSync({ publisher, subscriber })`

The realistic multi-region backplane, over Redis pub/sub. Bring your own
[ioredis](https://github.com/redis/ioredis)-compatible client (two connections —
`SUBSCRIBE` blocks its connection):

```js
import Redis from "ioredis";
import { redisSync } from "crossws/sync";

const ws = nodeAdapter({
  hooks,
  sync: redisSync({ publisher: new Redis(), subscriber: new Redis() }),
});
```

## Writing a driver

A driver is two methods. crossws calls `publish` for every local publish, and
calls your `deliver` callback for every message from another instance:

```ts
import type { SyncAdapter } from "crossws/sync";

const mySync: SyncAdapter = ({ id }) => ({
  // `id` is a stable per-instance id — use it to drop your own echoed messages.
  subscribe(deliver) {
    backplane.on("message", (raw) => {
      const { from, msg } = decode(raw);
      if (from !== id) deliver(msg); // {namespace, topic, data}
    });
  },
  publish(msg) {
    backplane.send(encode({ from: id, msg }));
  },
});
```
